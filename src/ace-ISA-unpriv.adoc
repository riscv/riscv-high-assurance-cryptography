[[ACE-unpriv]]
== ACE Unprivileged Architecture

=== Introduction

ACE relies on certain fundamental concepts, which are introduced here. While
this leads to some repetitions, this also brings better clarity to the
presentation.

[[ACE-concepts]]
==== Fundamental Concepts and Definitions

Cryptographic Context:::
(((Cryptographic Context)))
The *Cryptographic Context* (CC) is the foundation of ACE. A CC is an indivisible structure consisting of a *Metadata* header and a *Content* section; all cryptographic operations performed by ACE execute within the constraints defined by a CC. The Metadata has a fixed format and specifies the cryptographic algorithm and various optional policies associated with the CC.
(((Cryptographic Context, Metadata)))
The Content has an algorithm-specific format and holds sensitive material such as keys and internal algorithm state.
(((Cryptographic Context, Content)))
The Metadata requires integrity; the Content requires both integrity and confidentiality. These requirements are architecturally enforced by ACE and by the cryptographic algorithms it implements.

Context Register:::
(((Context Register)))
To create CCs, ACE uses *Context Registers* (CRs), per-hart architectural containers that safeguard their contents. A CR is *provisioned* by writing Metadata and Content into it; this operation binds the latter to form a CC.
There is no store-like instruction to extract the Content of the CR in a manner that compromises its confidentiality. A CC may be used for cryptographic operations only while resident in a CR. Only the ACE unit may access or modify a CC residing in a CR, in accordance with architectural rules and the algorithm bound to the CC.

Sealed Cryptographic Context:::
(((Cryptographic Context, sealed)))
To support context switching while at the same time protecting CCs outside architectural state, a CC may be *exported* from a CR as a *Sealed Cryptographic Context* (SCC), an encrypted and authenticated representation of a CC that can later be *imported* into a CR. SCC export and import use a _Context Sealing Key_ (CSK), which may be programmable by M-mode.
A CC sealed under one CSK cannot be re-imported under a different CSK, enabling cryptographic domain separation across spatial and temporal domains.

Object-Orientation:::
(((Cryptographic Context, object-oriented)))
CCs act as *objects* exposing a uniform interface across algorithms.
A CC encapsulates algorithm-specific behavior, enabling interchangeable use of different primitives (e.g., AES-128, AES-256, SM4) in identical modes (e.g., ECB, GCM-SIV, XTS) without requiring changes to compiled code.
Side-channel attack (SCA) resistant variants are supported transparently.

Usage Control Policy:::
(((Cryptographic Context, usage control)))
(((Cryptographic Context, management operations)))
(((Cryptographic Context, usage operations)))
ACE distinguishes between CC/CR *management* and *usage* operations.
(((Cryptographic Context, export)))
Management operations include *configuration* (e.g., CR provisioning, SCC import, and cloning) and *export* of a CR as an SCC. Usage operations include cryptographic operations and state changes.
(((Cryptographic Context, access)))
All operations represent a form of *access* to CRs.
To prevent denial-of-service scenarios, management operations, in particular configuration operations, are far less restricted than usage operations. _Usage Control_ policies restrict only certain CC/CR usage operations.

Localities:::
(((Cryptographic Context, Locality)))
ACE can bind CCs to a *Localities*, i.e., restrict usage to specific devices or device classes, hardware configurations, physical or virtual boot cycles, and process domains at various privilege modes. This binding is achieved by using the Localities as _tweaks_ in the encryption and decryption of SCCs.

[[ACE-architectural-model]]
==== Architectural Model

ACE is architected as an _attached_ unit, meaning that ACE instructions act as messages across a tightly coupled coprocessor interface.
The ACE unit receives the operations from the hart, executes them independently of the issuing hart, and returns results.

Beside implementations of ACE within a CPU core, this model also allows a single hardware instance of the ACE unit to be shared by multiple cores, with separate architectural states for each client hart.
// The specification mandates only behavior, not implementation.

The ACE unit may lock certain architectural states—e.g., Vector Registers, CRs, and CSRs.
The hart stalls when attempting to read them until the ACE operation using those resources completes.

This design allows a certain level of concurrency.
Long-running ACE operations may proceed even if the hart is preempted.
//For example, consider a U‑mode process executing an ACE instruction.
In this case, when the context‑switch handler accesses a locked register, different behaviors are allowed:
the hart may wait for the operation to finish;
the implementation may adopt a precise exception model, whereby the ACE unit is halted and sets `acestart` to allow later resumption;
or the system may impose a timeout before halting.
The architecture permits all these approaches.
It only specifies the allowed interruption points for its instructions and algorithms.

Conceptually, for each of the 32 CRs, ACE maintains a full 64-bit Metadata Header and an offset into internal memory pointing to the CR's Content (if not empty). The Metadata Header indicates whether a CR is configured. The actual representation of CRs in a concrete implementation may differ significantly.

=== Programmer-Visible Architectural State

[[ACE-cryptographic-registers]]
==== Context Registers

{empty}
(((Cryptographic Context)))
ACE defines 32 Context Registers (CRs), numbered 0 to 31 and denoted `K0`–`K31`.

CRs are architectural state, i.e., per-hart, like the GPRs or the vector registers.

Unlike a conventional register, a CR can be _unconfigured_, meaning it does not contain a CC.

A CR is addressed directly, as a 5-bit immediate, or indirectly, via a GPR value.
An additional instruction-encoding bit is used to select how the CR is addressed.
Accessing a CR with an indirect index outside [0..31] raises an illegal instruction exception.

An implementation must wait before accessing a CR that is still being written to asynchronously.

[NOTE]
====
Indirect CR addressing also an important microarchitectural implication.
Supporting CR renaming is possible but would significantly increase microarchitectural complexity.
CRs are redefined far less frequently than GPRs or FP registers (excluding implicit internal state changes).
Hence, CR renaming is not expected to significantly affect performance.
====

(((Context Register File)))
CRs reside in a dedicated, per-hart _CR file_ (CRF) accessible only to the ACE implementation. This memory shall guarantee
confidentiality and integrity of its contents, including the inability for an attacker to roll-back the latter to a previous state.
We do not require that accesses to this memory are hardened against physical side-channel attacks at the hardware level,
however, the implementation of side-channel protected algorithms in the ACE unit shall provide resistance to such attacks as well.

The amount of CRF capacity consumed by each CR depends on the algorithm encoded in the metadata, and it may differ from one CR to another.

Insufficient residual CRF capacity may prevent execution of a configuration instruction (`ace.prov`, `ace.import`, `ace.clone`):
Capacity is freed by unconfiguring other CRs.

The CRF capacity is implementation-defined. Implementations shall provide sufficient
capacity for all supported operations. At a minimum, the CRF must hold the
largest possible single CR, or the largest CR pair used in cloning operations.
An optional, implementation-specific mechanism for estimating required capacity
is described in <<ACE-instruction-size>>.

Any corruption of the CRF associated with a hart shall cause an unrecoverable error condition.
The entire architectural state of the ACE unit associated with the hart is reset, and exception `ace_guru_meditation` is raised (cf. <<ACE-exceptions>>).

We briefly summarize the most important operations on CRs, in order to be able to refer to them before they are fully defined.

* `ace.prov` (<<ACE-instruction-init>>) loads metadata and plaintext Content into a CR, binding them together.
* `ace.clear` (<<ACE-instruction-clear>>) unconfigures and clears a CR.
* `ace.export` (<<ACE-instruction-export>>) exports a CR to memory as an SCC.
* `ace.import` (<<ACE-instruction-import>>) imports a SCC into a CR.
* `ace.exec` (<<ACE-instruction-exec>>) performs cryptographic operations as specified by the algorithm configured in a CR and according to the state of the CR.
* `ace.setst` (<<ACE-instruction-setst>>) transitions between states of a cryptographic algorithm.
//* `ace.getmd` (<<ACE-instruction-getmd>>) reads the metadata of a CR.
* `ace.clone` (<<ACE-instruction-clone>>) creates a copy of a CR into a second CR.

// ////////////////////////////////////////////////////////////////////

==== Inputs and Outputs, RVV-mini

ACE inputs and outputs may use vector registers or an architectural input/output buffer (ACEIOBUF)
that can be loaded from and stored to external memory. Both mechanisms may be supported simultaneously.

When vector registers are used, the full V extension is not required. ACE depends only on a minimal subset, termed _RVV-mini_.
(((RVV-mini)))
RVV-mini must be fully opcode-compatible with RVV: code written for RVV-mini must execute with identical behavior under RVV. The required features are:

* Vector registers wide enough to support the inputs and outputs of the implemented algorithms;
* The `vsetvli` instruction with restricted parameters (e.g., `vsetvli zero,4,e32,m1,ta,ma` for 128-bit operations);
  unsupported configurations must raise an illegal instruction exception;
* Unit-strided vector loads and stores; and
* Vector bitwise logical operations: `vxor`, `vand`, and `vor`.

ACE instructions (currently) do not use vector masking and do not define masked versions of `ace.exec`.
Therefore, the algorithms currently defined require unmasked inputs and outputs.

Floating-point, fixed-point, and general arithmetic vector operations are not required.
Complex operations for certain cryptographic algorithms may be performed in GPRs, with data inserted into vector registers and result
extracted from them via `vins`/`vext`.

[WARNING]
RVV-mini need not be finalized before ACE ratification begins; it
may be defined later, and the ACE specification amended accordingly.
This allows additional requirements to be collected, for example, whether to mandate
`vclmul.v[vx]` and `vclmulh.v[vx]` from `Zvbc`.

The optional architectural, per-hart input/output buffer (ACEIOBUF) may be
used in place of vector registers, as described in <<ACE-usage-input-output>>.
The ACEIOBUF is not part of any CR.

The total length and the range of the ACEIOBUF used in cryptographic
instructions are controlled by the `aceiobuflen` (<<ACE-CSR-aceiobuflen>>) and
`aceiobuftop` (<<ACE-CSR-aceiobuftop>>) CSRs.

// Some instructions will accept an input and an output in the ACEIOBUF.
// The input and output may be identical or different, but there must be either
// identity or separation, partial overlapping is not allowed.
// Currently, everything is done in-place.

The ACE unit is agnostic to `VL` and `SEW`, but not to their product `VL*SEW` (or the length `aceiobuftop` of the ACEIOBUF in bits).
This means that while the CPU instruction-decoding side of ACE will use `VL` and `SEW` to determine the amount of data to send to the “ACE Unit”
side of the implementation, the latter will need only to know the product `VL*SEW` (or `aceiobuftop`) for its functionality.

[[ACE-Algorithm-field]]
====== _Algorithm_ and _AlgorithmPolicy_ Fields

Algorithms are encoded in an 11-bit field. Values 0–1535 and 2047 are
architecture-defined; the list of encodings is maintained by RVI (see
<<ACE-algorithms>>). The remaining values are available for custom algorithms.

For encryption/decryption primitives and modes, the _AlgorithmPolicy_ field
indicates whether the CC may be used for encryption (lower bit set), decryption
(upper bit set), or both. At least one of the bits must be set.

For asymmetric primitives supporting signatures, the lower bit permits
signature generation and the upper bit permits verification.
Such an asymmetric primitive can be configured to support
primitives such as key agreement or key encapsulation but _not_ signatures,
and in this case both bits are zero.

For other primitives, the _AlgorithmPolicy_ field may encode additional
algorithm identifiers to expand the effective encoding space of the _Algorithm_
field.

[[ACE-State-field]]
====== _State_ Field

The _State_ field is used to track the currently state of an algorithm, including certain types of errors.

The architected values of _State_ which are common to all algorithms are defined in
<<ACE-states-valid>>.

[[ACE-states-valid]]
.Global CC State Numbers: Valid States
[float="center",align="center",width="100%",cols="^7%,<27,<15%,<51%",options="header"]
|===
| Value | Mnemonic                  | State Name     | Description
|   0   | `ace_state_off`           | _Off_          | The CR is not configured.
|   1   | `ace_state_initial`       | _Initial_      | The state of a CC immediately after provisioning; ready for operation.
|  21   | `ace_state_success`       | _Success_      | Algorithm completed successfully. No further cryptographic operations may be performed until the CC is reset.
|  22   | `ace_state_failure`       | _Failure_      | Failure in an algorithm, usually an authentication failure in AE(AD) algorithms or a signature verification failure. No further cryptographic operations may be performed until the CC is reset.
|===

[[ACE-states-error]]
.Global CC State Numbers: Interrupted and Error States
[float="center",align="center",width="100%",cols="^7%,<27,<15%,<51%"]
|===
|  23   | `ace_state_partial_config`  | _Partial_Config_   | Provisioning or import interrupted.
|  24   | `ace_state_expired`         | _Expired_       | The CR has expired.  This is set by the hardware even if the condition also raises an exception.
|  25   | `ace_state_import_auth`     | _Auth_Failed_   | Authentication failure with `ace.import`.
|  26   | `ace_state_priv_viol`       | _Priv_Viol_     | Violation of Usage Control policies.
|  27   | `ace_state_out_of_mem`      | _Out_of_Memory_ | Insufficient CRF memory for configuration instruction. This may be set by privileged code in case they do not handle the CR allocation.
|  28   | `ace_state_not_impl`        | _Not_Impl_      | Not implemented algorithm. It is set by the hardware upon an attempt to provision or import a CR that is not supported by the implementation.
| 29, 30 | (Reserved)                 |                 | Reserved for future use.
|  31   | `ace_state_invalid`         | _Invalid_       | Invalid/invalidated state, for instance because of the use of an invalid value (argument, state, or metadata).
|===

The state with value 0 is termed _Off_, and is used to indicate that the CR is not configured.
The Metadata Header in this case is the all-zeros value `zeros(64)`.

All states in the range 1-22 are termed _valid_. This includes both _Success_ and _Failure_ states, which must be used for
normal termination of an algorithm in the cases of its successful completion or failure.

State 23, i.e., _Partial_Config_, is used to report that provisioning or import was interrupted.

States 24-31 are termed _error states_, and are used to report certain specific ACE errors.
(((ACE, algorithm, invalid state)))
(((Context Register, invalidation)))
In particular, if the State is `ace_state_invalid`, we say that the CR is _invalidated_.

An error state can be set by software using `ace.setst` (with no auxiliary input).

Error conditions that lead to exceptions are treated in <<ACE-exceptions>>.

If an algorithm defines 22 or fewer valid states (including _Initial_, _Success_, and _Failure_), the current state is stored in the _State_ field.
For algorithms with more than 22 valid states, additional state information resides in the Content section.

The following rules are generic and are valid for all algorithms:

. The Metadata's _State_ field must reflect the current state whenever the CC is in _Initial_, _Success_, _Failure_, or any error state.
This ensures that these states can be unambiguously determined from the _State_ field alone, without inspecting the Content.

. In States _Success_ and _Failed_, only the following ACE instructions are permitted: `ace.prov`, `ace.import`, `ace.export`, `ace.clear`, `ace.size`, `ace.setst`, and `ace.getst`.  From these two states, `ace.setst` is only permitted to transition to _Initial_ (or _Off_ if used as `ace.clear`).

. Unless an algorithm's specification explicitly states otherwise, a transition from any valid state to _Initial_ using instruction `ace.setst` is always permitted.

. Upon transitioning to an error state, the Content of a CR is invalidated and emptied.
Internal resources associated with the Content may also be released. In that case, any Additional Data is released and the _AddDataSize_ field is set to 0.
The CR then retains only the Metadata Header.

. Exporting a CR in an error state exports only the metadata (and the authentication tag).
  Requesting the SCC's size (`ace.size` <<ACE-instruction-size>>) wil reflect this.
  The resulting SCC can be reimported, allowing software to take appropriate actions even after a context switch.

. If a CR is in any error state, it can still be reconfigured, cloned, or cleared. In particular, `ace.setst` can only transition it to _Off_.

. The metadata of a CR in an error state can be read (using `ace.getmd`: <<ACE-instruction-getmd>>).

. Attempts to _use_ a CR in any error state will not perform any operation, and in particular they leave the _State_ field unchanged.

. If a CR is in State _Partial_Config_, transitioning it to any State other than _Off_ or _Invalid_ is only possible as a consequence of a restart or resumption of the interrupted configuration instruction.  `ace.setst` can only transition it to _Off_ or _Invalid_.

. Any _usage_ instruction that would violate the Usage Control policies will not perform its operation, and will set _State_ to `ace_state_priv_viol` instead.

We note that the State after `ace.setst` is issued may differ from the State specified by the instruction.
A common example is MAC verification: after `ace.setst` is issued with an `ace_state_hash_verify` immediate as the next state and the value to be verified as the additional parameter, the algorithm transitions to either _Success_ or _Failure_. This is done to prevent the manipulation of a CR's state and to reduce code size.
In certain circumstances, also executing an `ace.exec` instruction may trigger a state change.
Many examples of such state changes are listed in <<ACE-algorithms>>.

[[ACE-Localities]]
====== _Locality_ Field

ACE maintains a table of 128-bit Locality Secrets (LS), the Locality Secrets
Table (LST). Some entries are fixed or configurable only via an
implementation-specific authenticated hardware procedure; these are global and
shared across all ACE units in the same SoC. Other entries are per-hart
architectural state programmed by privileged modes.

<<ACE-locality-indexes>> lists the architected Localities. Eleven LST entries
are defined: the first six constitute the _HW Binding Group_, binding SCCs to
the hardware, entries 6–7 form the _Boot Session Group_, and entries 8–10 form
the _SW Filter Group_. Seven of the eight _Locality_ field bits are defined;
bit 7 is zero when Localities are defined as in <<ACE-locality-indexes>>.

For each Locality, the following information is provided: index in
the LST; domain (SoC-, device-, VM-wide, or per-process); configurability;
bit-field encoding and value.

[[ACE-extra-content]]
====== _AddDataSize_ Field

The _AddDataSize_ Metadata field records the length of the _Additional Data Section_ of the content, which is a variable-size part which may
depend on the state of the algorithm.

The format of the _Additional Data Section_ is described in <<ACE-data-formats>>.

The first eight bytes of the Implementation-Specific Additional Data Section of the Content, if not empty, always hold a copy of the `acemarchid` CSR (<<ACE-CSR-acemarchid>>) of the originating machine, and are counted within the length recorded by the _AddDataSize_ field.
The importing implementation uses this value to determine whether any information in the Implementation-Specific Data Section can be used.

It is the responsibility of the *algorithm* to determine the lengths of the two parts of the Additional Data Section.
If the Implementation-Specific Additional Data Section is not empty, the *implementation* will check the value of `acemarchid` stored therein to decide whether the Implementation-Specific part can be used or should be ignored.

[NOTE]
====
Examples of Additional Data are:

. “Decryption AES keys”, i.e., the last one or two AES round keys, from which one can work backwards when performing AES decryptions.
. Precomputed point multiples for elliptic curve scalar multiplication (their computation can be far more expensive than their
encryption and decryption);
. Information, such as additional elliptic curve points or random numbers used by Elliptic Curve algorithms only in certain states;
. any auxiliary data for side-channel-protected implementations, such as key shares or other data to accelerate share reconstruction.

Examples 1., 2, and 4. are optional data, and therefore should be stored in the _Implementation-Specific_ Additional Data Section.
The third example, however, describes information that is required for the correct functioning of the algorithm. Therefore, its format must be defined by the algorithm specification and the data should be stored in the _Architected_ Additional Data Section.
====

// ////////////////////////////////////////////////////////////////////

[[ACE-Metadata-expiration-date]]
====== _ExpirationDate_ Field

If non-zero, _ExpirationDate_ specifies the CC expiration time as a number of
hours elapsed since the base date of January 1, 2027. The 18-bit field provides
for a span of approximately 30 years (2^18^ hours). Future 128-bit metadata formats
(encoded with _Algorithm_ = `ones(11)`) may support expanded time spans.

[[ACE-data-formats]]
===== Provisioning Input, Sealed Cryptographic Context, and Content

The format of a Provisioning Input (PI) is shown in <<ACE-PI>>.

[[ACE-PI]]
.Format of the Provisioning Input (PI), All Sections in Order
[width="100%",cols="28%,44%,28%",options="header"]
|===
| Section                      | Size | Description
| Metadata Header              | 64 bits | Contains the metadata information as described in <<ACE-Metadata>>.
| Provisioning Content         | Depends only on the _Algorithm_, _AlgorithmPolicy_, and _KeyType_ fields of the Metadata Header. Always a multiple of 64 bits. | Contains keys and other algorithm-specific information.
|===

The length of a PI must be a multiple of 64 bits.

The CR itself may contain significantly more information than the Provisioning Content.
This information may be categorized under different classes:

. Information that is necessary for the algorithm to function, and cannot be reconstructed from other information.
For instance, random values that are generated in the _Initial_ state.

. Information that may be easily reconstructed from other information in the State, so that there is no need to export it.
In this case, the *algorithm* may choose to not include it into an SCC.

. Data that can be used to optimize performance on a specific *implementation*, and that is more expensive to recompute than encrypting and storing externally. The hardware may choose to export such data in a custom format inside an SCC for efficient reuse.
A different implementation that receives the same SCC (i.e., with a `acemarchid` mismatch) may safely ignore this data during import.

The format of a Sealed Cryptographic Context (SCC) is shown in <<ACE-SCC>>.

[[ACE-SCC]]
.Plaintext Sealed Cryptographic Context (SCC) Format, All Sections in Order
[width="100%",cols="28%,44%,28%",options="header"]
|===
| Section                       | Size | Description
| Metadata Header              | 64 bits | Contains the metadata information as described in <<ACE-Metadata>>.
//
| Content: _Fixed Data Section_
| Empty if the context is in an error state, otherwise it depends only
on the _Algorithm_, _AlgorithmPolicy_, and _KeyType_ fields of the Metadata Header.
 Always a multiple of 64 bits.
| Contains keys and other algorithm-specific information.
//
| Content: _Architected Additional Data Section_ (Architected ADS)
| Empty if the context is in an error state, otherwise it depends on the information contained in the Metadata and Fixed Data Sections.

The second and third parts together of the Content form the _Additional Data Section_ of the Content.

The Metadata _AddDataSize_ field specifies the total size of the _Additional Data Section_. It does not partition this total between the Architected and Implementation-Specific components. The ordering of sections within the SCC allows to determine the lengths of these two components before reading them.

If a configuration instruction cannot be performed because of insufficient residual CRF memory, exception `ace_exc_out_of_mem` is raised. System software is responsible for freeing internal ACE unit space and retrying the instruction. Alternatively, the software may set the CR's State to `ace_state_out_of_mem` and return from the exception handler.

When a non-configuration instruction is executed on a CR whose _ExpirationDate_ lies in the past, the hardware immediately sets the Metadata _State_ field to `ace_state_expired` and performs any action prescribed for error states, including implementation-specific tasks such as releasing the CRF capacity held by the Content.  If the instruction is a usage instruction, the operation is suppressed and exception `ace_exc_expired_CR` is raised.  Exporting, cloning, or configuring an expired CR is allowed.

When an expired CR is used, privileged code may take appropriate action—for example, provisioning a new key or returning to the caller for handling.

Since privileged code may replace expired CRs, any user code that uses CRs that are known to potentially expire should always export the CR when it has finished using it, in case it may be needed later, instead of recovering it from a previously saved copy.

[[ACE-CSR-acestart]]
===== `acestart`

{empty}
(((CSR, `acestart`)))
`acestart` is an alias for `vstart`. Therefore, it is an XLEN-bit RW CSR.
ACE defines it also when the V extension is not enabled.

`acestart` tracks the progress of interruptible ACE instructions, namely
`ace.prov`, `ace.import`, `ace.export`, `ace.input`, and `ace.output`. It also
applies to `ace.exec` when iteratively processing a multi-block operand, as
`ace.exec` may be interrupted between blocks. Only these instructions honor a
nonzero `acestart`; all other instructions ignore it and are restarted, with
`acestart` also being cleared. Provisioning, import, and export are
resumed only under certain conditions, as explained in
<<ACE-provisioning-export-import>>.

`acestart` records the progress of an interrupted
operation. Since ACE does not have an _intrinsic_ concept of element length,
`acestart` counts the number of bytes processed so far, not the
number of elements.
//The top 8 bits of `acestart` are used to encode additional
//information and are ignored if not used.

// `acestart`[XLEN-1:XLEN-8] is non-zero only if the interrupted instruction is
// `ace.prov`, `ace.import`, or `ace.export`, in which case it is equal to
// `acestart`[XLEN-1:XLEN-8] = `3'b100 @ bin(n,5)` where `n` is the number of the
// CR being provisioned, imported, or exported at the time of interruption.

Hardware writes this register on interruption.
Software may save and restore the register across context switches.
For certain algorithms, writing zero to `acestart` to force a full restart
of `ace.exec` may lead to inconsistent or unpredictable results.
Writing an invalid or non-supported nonzero value raises
an illegal instruction exception.

When a vector instruction is interrupted, the CSR follows `vstart` semantics as expected,
i.e., it counts the number of processed elements.
//When an ACE instruction is interrupted, the CSR follows the `acestart` semantics described here.

[WARNING]
The reason `vstart` can be reused as `acestart` comes from the fact that the Risc-V ISA exposes a computation model in which the observable results are the same as if the instructions were executed in order.
If an implementation has executed instructions past the PC when an interrupt occurs, those instructions are discarded and must be re‑executed upon return from the handler. Only the instruction at the PC may be interrupted in a way that it saves temporary state that allows it to be resumed later.
There is no situation in which the microarchitecture of a hart can simultaneously interrupt multiple instructions allowing them to resume at the same time.
Hence, `vstart` can safely be reused by ACE as `acestart`.

{empty}

`aceiobuflen` is an XLEN-bit URW CSR that programs the ACEIOBUF length for use

except as part of the entire ACE state in `*status`.
We recommend software to set `aceiobuflen` to zero once the contents of the ACEIOBUF are no longer needed.
Context switching code can use this information to avoid unnecessary save and
restore of ACEIOBUF contents.

[[ACE-instructions-detailed]]
=== Instructions

[WARNING]
====
The encodings presented here are preliminary.
====

ACE provides a number of unprivileged instructions.
This section specifies their semantics and encodings.

The notation `K(Xd)` denotes the CR whose index is held in GPR `Xd`. `Kn|K(Xn)`
indicates two alternative encodings: a 5-bit immediate CR index (`Kn`) or a CR
number supplied by GPR `Xn`. Use of the zero register `X0` as a CR index
register raises an illegal instruction exception unless explicitly indicated
otherwise.

Each instruction is characterized by the following properties:

* _Modifies state_: whether the instruction may modify ACE architectural state;
* _Usage-controlled_: whether the instruction is subject to usage control
  policies (_UsagePolicy_);
* _Trappable_: whether the instruction may be implemented by privileged code
  via trap-and-emulate; and
* _Interruptible_: whether execution may be interrupted (e.g., by a context
  switch or page fault) and later resumed or restarted.

NOTE: Trappability refers solely to the CPU-side implementation—for example, to
enable decomposition of certain ACE instructions into smaller custom
operations. This is distinct from custom algorithm implementations within the
ACE unit itself (e.g., via a secure embedded CPU with updatable firmware),
which are outside the scope of this specification.

NOTE: ACE does not have concepts of
Element Group Width, Effective Element Width,
and
Element Group Size like the vector crypto extension.
Because of the fact that the architecture decouples instructions like `ace.exec` from the
actual operations, the high level architecture of ACE only needs to know the total size
of the inputs and output, which is `ACELEN`, namely either `VL*SEW` or the length
`aceiobuftop` of the ACEIOBUF as defined in <<ACE-notation>>.

[[ACE-instruction-init]]
==== ace.prov

_Synopsis_::
Provision Metadata and secrets into a CR.

_Mnemonic_::
`ace.prov Kd|K(Xd), %offset(Xs)`
(((ACE, instruction, ace.prov)))

_Encoding_::
`ace.prov` is an I-type instruction.
+
[wavedrom, , svg]
....
{reg:[
    { bits:  7, name: 0x0b, attr: ['custom-0'] },
    { bits:  5, name: 'context', attr: ['Kd','K(Xd)'] },
    { bits:  3, name: 'ace.prov', attr: [0x4,0x5] },
    { bits:  5, name: 'Xs' },
    { bits: 12, name: 'immed12', attr: ['offset'] },
]}
....

_Description_::
`ace.prov` provisions a CR with Metadata and Content to create a CC. `Kd` or
`K(Xd)` denotes the CR to be provisioned. `Xs` points to the region of memory
containing the Metadata and the key material. The PI must be stored at an
8-byte aligned address, and its length must be a multiple of 8 bytes.
+
If the instruction fails, the target CR is unconfigured, with the same
semantics as if `ace.clear` were invoked on the same CR
(<<ACE-instruction-clear>>).
+
If the combination of _Algorithm_,
_AlgorithmPolicy_, and _SCProtection_ is not supported, the hardware simply
configures the CR in error state `ace_state_not_impl`.
+
Exception `ace_exc_invalid` is raised if the Metadata is invalid (other
than unimplemented algorithm), such as, for instance: a nonzero _State_ or
reserved field; or an unsupported _Locality_ value.
The handler may simply set the CR to error state `ace_state_invalid` and
return to the original control flow.
+
Similarly to vector store instructions, `ace.prov` appears to other harts as a
sequence of individual reads.
+
The operation is described in detail in <<ACE-provisioning-export-import>>.

_Properties_::
Possibly modifies state: *YES*. +
Usage-controlled: *NO*. +
Trappable: *YES*. +
Interruptible: *YES*. +

* `Xs` follows the format of the Metadata Header.
Only the fields _AlgorithmPolicy_, _SCProtection_, _Locality_, _UsagePolicy_,
and _ExpirationDate_ of `Xs` may be defined. All other fields must be 0, else
exception `ace_exc_invalid` is raised.

* `Xs`._AlgorithmPolicy_ is used to set the new `mh`._AlgorithmPolicy_ of the

* A one in any of the three bits `Xs`._UsagePolicy_ means that usage by the

  of the destination CC is set to 1.

* If `Xs`._ExpirationDate_ is zero, the corresponding policy in
  `mh`._ExpirationDate_ is _not_ changed.

* A non-zero value in `Xs`._ExpirationDate_ will be copied to
  `mh`._ExpirationDate_, but only if that policy was not already programmed before
  or the original value of `mh`._ExpirationDate_ was equal or larger.
  Otherwise exception `ace_exc_invalid` is raised.

* `s` must be even, the register pair `X[s]` and `X[s+1]` are read, and

  Metadata section given in `Xn` into GPR `Xd` including the Additional Data,

  supported, and `NOT(bin(ace_state_invalid, XLEN))` if the Metadata is invalid.

  be allocated, and the destination register contains the size of the buffer in

NOTE: In general, no correlation may be assumed between the size of a PI, the
corresponding CR's usage of internal resources, and the size of the SCC.

[[ACE-interruptibility]]
=== Interruptibility of ACE instructions

`ace.clear`, `ace.reset`, `ace.setst`, `ace.getst`, `ace.restrict`, `ace.clone`, `ace.size` and `ace.avail` are uninterruptible.

`ace.import`, `ace.export`, `ace.prov`, `ace.input`, and `ace.output` can be interrupted.
The interruptibility of provisioning, import, and export operations is
defined in <<ACE-provisioning-export-import>>.
For `ace.input` and `ace.output`, interruptibility limited to interrupts and memory errors.
`acestart` will reflect the amount of data read or written before the interrupt and resumption will start from that point.

The operation of `ace.exec` on a basic processing unit is uninterruptible,
unless explicitly allowed by the algorithm. In general, most cryptographic
primitives cannot be interrupted; therefore, state information to encode
round number or round key value is not needed.  For certain very long running
operations, such as public key primitives, an algorithm specific
interruptibility behavior is defined.

When `ace.exec` processes multiple blocks iteratively, each block is handled atomically. The instruction may be interrupted between blocks. Upon receiving an interrupt during block processing, the implementation has two options:

* Complete the current block before yielding to the interrupt handler. `acestart` points to the next block.
* Discard the partially executed block and restore the state prior to that block's processing. `acestart` points to the partially processed block.

[[ACE-provisioning-export-import]]
=== Provisioning, Export and Import of Cryptographic Contexts

In this section we implicitly refer throughout to <<ACE-data-formats-intro>>, and in particular to <<ACE-PI>> and <<ACE-SCC>>.

The entire Metadata and Content are authenticated, but only the Content is encrypted.
The AEAD method used for this purpose is described in <<ACE-SCC-AEAD>>.

In an algorithm specification (<<ACE-algorithms>>), under Data Structures there is a section named “Serialized Content”: It defines the structure of the `content[]` array.  Architecturally, `content[]` is the serialization of the content of a CR for export as an SCC.
This array is this the concatenation of the Fixed Data Section and of the Architected and Implementation ADSs.

[NOTE]
--
While `content[]` provides a view of (certain fields of) the content of a CR
that matches the serialized SCC format, it need not be converted to and from
the internal representation and explicitly stored in the ACE unit's memory
before export and after import.

`content[]` is a virtual representation: its byte ordering does not imply
physical contiguity within the CR, nor is there a direct byte-for-byte
correspondence. Changes to `content[]` are reflected in the underlying CR
representation (except for excluded fields), and vice versa.

In a threshold implementation, a single key byte in `content[]` may correspond to multiple bytes distributed across all key shares within the CR.
Reading that byte returns the XOR of the corresponding bytes from the shares, which may not be stored contiguously.
Writing to that byte may trigger multiple writes of single bytes, one per share, such that their XOR equals the input.
Alternatively, these operations may be deferred until the end of a provisioning or import instruction.
For example, writing to `content[]` may initially write the unmasked value to the first share; the remaining shares are generated only when the instruction completes.
--

[[ACE-system-keys]]
==== System Keys and Randomly Generated Key Values

In a PI or SCC, System Keys are encoded as a 64-bit index into an internal, system-specific System Keys Table (SKT). This replaces the key value itself, which may reduce the size of the PI or SCC content.

For threshold implementations of system keys, ACE may copy key material from the SKT into the CR and expand it into multiple shares. The key is still exported as a single index.

For the purpose of accelerating key share reconstruction in a threshold implementation, some shares of a system key can be included in the content's Additional Data, as long as the key cannot be recovered from those shares alone.

An all-ones (i.e., `ones(64)`) system key index in a PI means that the key material is generated randomly upon provisioning.
The _KeyType_ field in the metadata is then internally set to 0, and the key is then exported by value.
This applies to all keys at once in an algorithms, unless otherwise specified.

[[ACE-size-functions]]
==== Notation for the Size Functions

Let `mh` denote a Metadata Header.
In order to determine the total lengths of the PI and of the SCC, the implementation provides the following functions or an equivalent mechanism:

. ``MH_size(mh)`` returns the byte length of the Metadata Header. In the
  current specification, it is always 8.

. ``PI_content_size(mh)`` returns the byte length of the content section of a
  PI. It depends only on `mh`._Algorithm_, `mh`._AlgorithmPolicy_, and
  `mh`._KeyType_.

. ``fixed_content_size(mh)`` returns the byte length of the fixed part of the
  content section of an SCC. It depends only on  `mh`._Algorithm_,
  `mh`._AlgorithmPolicy_, and `mh`._KeyType_. It does not include the length of
  the Additional Data of the Content, of length _AddDataSize_ * 8 bytes.

The total size in bytes of a PI is therefore

`{fournbsp}{fournbsp}len_in_bytes(PI){nbsp}` = `MH_size(mh)` + `PI_content_size(mh)`,

and the total size in bytes of an SCC is

`{fournbsp}{fournbsp}len_in_bytes(SCC)` = `MH_size(mh)` + 16 + `fixed_content_size(mh)` + _AddDataSize_ * 8.

[[ACE-interruption-points]]
==== Interruptibility of Configuration and Export Operations

During `ace.prov` and `ace.import`, _State_ is set to _Partial_Config_ for the
duration of the operation. It is updated to the correct value only upon
successful completion (see <<ACE-provisioning-export-import>>).

This transition to _Partial_Config_ does not occur during `ace.export`.

Provisioning and import instructions may resume from the point of interruption
only if control returns directly after the interrupt handler completes.
This is characterized by a CR with _State_ set to _Partial_Config_ and
a nonzero `acestart` value.
The privileged interrupt handler must guarantee restart only in this case.

A partially configured CR cannot be exported.
Therefore, at context save privileged code must clear a partially configured CR.
Before returning control flow to the process of the interrupted configuration operation, the CR must be cleared as well
to force the instruction's restart.

Provisioning, export, and import are interruptible only before any memory
access they perform. This means that interrupts other than memory exceptions
must wait until the next memory access, or can be served immediately provided
the CR is reverted to the last interruptible point.

Upon returning to the original control flow from an interrupt or exception,
an export may resume only if the CR has not been modified.
The mechanism to enforce this rule is described in <<ACE-SCC-export>>.

The rules for resuming configuration and export operations imply that these
can only occur on the same hart where they have been interrupted. In particular,
these resumptions can only occur on the same microarchitecture and therefore
the implementation can define arbitrary interruption points and corresponding
non-zero values of `acestart`.

[[ACE-SCC-AEAD]]
==== Key Derivation and POLYVAL

AES-GCM-SIV cite:[DBLP-journals-iacr-GueronLL17,RFC8452] is used to export and
import CRs.

[WARNING]
AES-GCM-SIV has been chosen because it only required single passes of data in
main memory, reducing the number of external memory accesses. GCM uses also a
single pass but requires separate nonce and tag values, increasing the size of
the exported data by one block.

We define the Key Derivation (<<ACE-SCC-RFC8452-derivation>>) and POLYVAL
(<<ACE-SCC-POLYVAL>>) functions first. The generation of an SCC from a CR
(<<ACE-SCC-export>>) and the import of an SCC into a CR (<<ACE-SCC-import>>)
are defined next.

[[ACE-SCC-RFC8452-derivation]]
.RFC8452 Key Derivation
[]
----
function RFC8452_KeyDeriv(key : bits(256)) = {
  struct {
    AESE256(key, (zeros(125) @ 0b101))[63:0] @ AESE256(key, (zeros(125) @ 0b100))[63:0] @
    AESE256(key, (zeros(125) @ 0b011))[63:0] @ AESE256(key, (zeros(125) @ 0b010))[63:0],
    AESE256(key, (zeros(125) @ 0b001))[63:0] @ AESE256(key, (zeros(125) @ 0b000))[63:0]
  }
}
----

NOTE: Due to the computational cost of the derivation functions,
implementations may cache recently used derived keys.

`POLYVAL` is a so-called Almost XOR-Universal (AXU) Hash Function.
The SIV is the encryption of the computation of this AXU hash on the concatenation of the Locality Secrets, the Metadata zero-padded to 128 bits, and the Content of the CR.

In the pseudocode below for POLYVAL, the data to be digested by POLYVAL is represented an array with an integral number of 16-byte blocks, where the last one may be just 8 bytes long.

[[ACE-SCC-POLYVAL]]
.Polyval Hashing Function
[]
----
function POLYVAL(&tmp : bits(128), auth_key : bits(128), blocks[0 .. n-1] : bits(128),
                 last_block_fractional : bool) = {
  tmp ← zeros(128);
  local i : int;
  foreach(i from 0 to n-1) {
    if ((i == n-1) and last_block_fractional) {
      tmp[63:0] ← tmp[63:0] xor blocks[i];
      tmp ← Montmul(tmp, auth_key);
    } else {
      tmp ← tmp xor blocks[i];
      tmp ← Montmul(tmp, auth_key);
    }
  }
}
----

[[ACE-SCC-export]]
==== Export

During SCC export and import, the Locality Secrets associated with the
Localities encoded in the _Locality_ field of the Metadata are prepended, in
ascending index order, to the input of the AEAD tag computation, before the
metadata. If the _Locality_ field is zero, no Locality Secrets are prepended.

The ACE architecture supports only one export/import operation per hart at a time.
Accordingly, an implementation will maintain only a single per hart instance of variables for `SIV` and `tmp`.
The implementation maintains information to track an export in progress:
a 1-bit flag `ace_exporting` and a 5-bit value `ace_exported_cr` recording the number of the CR being exported.
`ace_exporting` and `ace_exported_cr` are set when an export is started, and cleared when it is completed.
If the export is interrupted, `acestart` is set accordingly.
When this flag is set, any instruction that modifies the internal state of CR number `ace_exported_cr` clears `ace_exporting`.
Export may be resumed only if the flag is set and only for the recorded CR.
Otherwise the export is restarted and `acestart` is ignored.
Without this protection, resuming an export could produce an incorrect SIV for the remaining SCC data.

[[ACE-SCC-export-listing]]
.Generation of an SCC from a CR
[]
----
function ace.export(&CR, M[]) = {
  mh : bits(64) = & CR.Metadata;           // By reference, from the CR
  external CSK : bits(256);
  external enc_key : bits(256);            // RFC8452_KeyDeriv(CSK)
  external auth_key : bits(128);

  external SIV, tmp : bits(128);
  local z : int ← fixed_content_size(mh);  // can be recomputed upon resumption
  z ← z + mh._AddDataSize_ * 8;

  CR.content = array of z bytes; // includes the Additional Data
  // In external memory: Metadata Header, then SIV, then Content
  M = array of MH_size(mh) + 16 + z bytes.

  local let last_block_fractional : Boolean ← z % 16 != 0;
  local let num_blocks : int ← ceil(z / 16);
  local L : string of bits;

  // Start POLYVAL
  // Absorb Locality "prefix" (zero or more 128-bit blocks)
  tmp ← zeros(128);
  foreach(i from 0 to number of localities - 1)
    if (CR.Metadata.Locality includes locality[i]) {
      tmp ← tmp xor LST[i];
      tmp ← Montmul(tmp, auth_key);
    }

  // Absorb the metadata block into POLYVAL
  tmp ← tmp xor (zeros(64) @ mh);
  tmp ← Montmul(tmp, auth_key);

  // Absorb the entire content into POLYVAL
  foreach(i from 0 to num_blocks - 1) {
    if ((i == num_blocks - 1) and last_block_fractional)
      tmp[63:0] ← tmp[63:0] xor content[i*16 .. 7+i*16];
    else
      tmp ← tmp xor content[i*16 .. 15+i*16];
    tmp ← Montmul(tmp, auth_key);
  }
  // compute the SIV
  SIV ← AESE(enc_key, 0 @ tmp[126:0]);

  // Export metadata and the SIV to memory
  M[0 .. 7]  ← mh;
  M[8 .. 23] ← SIV;

  // Encrypt the content and store it in memory
  foreach(i from 0 to num_blocks - 1) {
    if ((i == num_blocks - 1) and last_block_fractional)
      M[24+i*16 .. 31+i*16] ← CR.content[i*16 .. 7+i*16]
               xor AESE(enc_key, 1 @ SIV[126:32] @ (SIV[31:0]+bin(i,32) % 2**32)[63:0];
    else
      M[24+i*16 .. 39+i*16] ← CR.content[i*16 .. 15+i*16]
               xor AESE(enc_key, 1 @ SIV[126:32] @ (SIV[31:0]+bin(i,32) % 2**32);
  }
}
----

// Optional Feature: Interruption of ace.export's SIV computation:::
// Normally, the first point of interruption of `ace.export` after the key derivation would be immediately before the first block to the input of `POLYVAL` is processed, resulting in a `acestart` value of 1.
// An implementation is allowed to either prevent `POLYVAL` to be interrupted or to interrupt it and restart its computation if `acestart` is equal to 1.
// A third option, namely the resumption of `POLYVAL`, is allowed: with reference to <<ACE-SCC-POLYVAL>>, it is possible to interrupt `POLYVAL`
// immediately after the completion of the `montmul` operation.
// In this case the value of `acestart` will be 16{interpunct}`i`+ 2, the number of bytes absorbed so far (which is always a multiple of 16), plus 2.

[[ACE-SCC-import]]
==== Import

The import algorithm needs some special care:

// * The first block read from memory by the operation is the Metadata Header
//   `mh`.
// ** If strictly fewer than four bytes are read before the instruction is
//    interrupted, ACE leaves `acestart` at 0, and the instruction will be
//    restarted upon resumption.
// ** If four bytes or more have been read, the value of `mh.State` in the
//    SCC is known. In this case, the instruction may be restarted WITH
//    `acestart` {ge} 4 recording the amount of data read.
* At the beginning of the import operation, the hardware saves the value of
  `State` from the original metadata and sets `mh.State <-
  ace_state_partial_config`. At the end of the import operation, `mh.State` is
  set to the saved value.
// ** Since context switching code cannot export a partially imported CR when
//    saving the state, it will clear the CR instead, forcing the hardware to
//    restart the import operation.
// ** Exceptions that immediately return to the interrupted operation, such as
//    memory exceptions, may cause the hardware to resume the operation instead,
//    if the CR was left untouched.
* The SCC to be imported may have been generated by a different implementation
  than the one importing it, and may contain an Implementation-Specific ADS
  that exceed the maximum size supported by the importing hardware for the
  algorithm encoded in the Metadata.
  In this scenario, once the Metadata and the Fixed Content Data Sections have been
  read, the implementation must determine the length of the Architected ADS
  corresponding to the algorithm's state, and ignore all writes to `content[]`
  beyond the end of the Architected ADS.
  It must still process the entire SCC to verify the authentication tag.
  After successful verification, the _AddDataSize_ field will be adjusted
  to the size of the Architected ADS.
  The Implementation-Specific ADS will be ignored.
  The details of this adjustment are not reflected in the pseudocode
  <<ACE-SCC-import-listing>> to avoid excessive complexity.

// [NOTE]
// ====
// Emulating parts of the `ace.export` instructions requires
// implementation-specific operations from `ace.exec`. Specifically, an
// implementation will have to define ACE operations to perform the decryption of
// the block, and write the result to a vector or general-purpose register. The
// contents of the vector or general-purpose register can then be stored using a
// standard store operation. Such operations can not be normative as they depend
// on how the ACE unit is implemented.
// ====

// *** The hardware will only resume the import operation if the flag is set,
// otherwise it will restart the operation.
// CR.importing is always false except during import.
// During import, the original State is saved and temporarily replaced in the CR
// with ace_state_invalid to prevent any USAGE operations.  Any operation that writes
// to the CR will set CR.importing to false: this allows context switching code to reuse
// the CR and, upon returning to the interrupted import, the hardware will detect
// CR.importing == false and restart the import instead of resume it.  OTOH, if the
// interrupt was caused by a memory exception, the instruction can be resumed.

[[ACE-SCC-import-listing]]
.Importing an SCC into a CR
[]
----
function ace.import(&CR, M[]) = {
  // M : array of bytes;   from memory
  local saved_SN : bits(4);                // This is a "hidden" state as well
  external SIV, tmp : bits(128);

  external CSK : bits(256);
  external enc_key : bits(256);
  external auth_key : bits(128);

  tmp[63:0] ← M[0 .. 7];         // read Metadata Header

  saved_SN ← tmp[63:0].State;
  CR.Metadata ← tmp[63:0];
  CR.Metadata.State ← ace_state_partial_config;

  & mh = CR.Metadata;

  z ← fixed_content_size(mh);
  z ← z + mh._AddDataSize_ * 8;

  if (z == 0)
    raise exception ace_exc_not_impl;

  if (there is not enough memory for the algorithm encoded in the Metadata)
    raise exception ace_exc_out_of_mem;

  num_blocks ← ceil(z / 16);
  last_block_fractional : Boolean ← (z % 16 != 0);

  // Start POLYVAL
  // Absorb Locality "prefix" (zero or more 128-bit blocks)
  tmp ← zeros(128);
  foreach(i from 0 to number of localities - 1) {
    if (CR.Metadata.Locality includes locality[i]) {
      tmp ← tmp xor LST[i];
      tmp ← Montmul(tmp, auth_key);
    }
  }

  SIV ← M[8 .. 23];
  tmp ← tmp xor (zeros(64) @ mh);
  tmp ← Montmul(tmp, auth_key);

  foreach(i from 0 to num_blocks - 1) {
    if ((i == num_blocks - 1) and last_block_fractional) {
      content[i*16 .. 7+i*16] ← M[24+i*16 .. 31+i*16]
          xor AESE(enc_key, 1 @ SIV[126:32] @ (SIV[31:0]+bin(i,32) % 2**32)[63:0];
      tmp[63:0] ← tmp[63:0] xor content[i*16 .. 7+i*16];
    } else {
      content[i*16 .. 15+i*16] ← M[24+i*16 .. 39+i*16]
          xor AESE(enc_key, 1 @ SIV[126:32] @ (SIV[31:0]+bin(i,32) % 2**32);
      tmp ← tmp xor content[i*16 .. 15+i*16];
    }
    tmp ← Montmul(tmp, auth_key);
  }
  tmp ← AESE(enc_key, 0 @ tmp[126:0]);

  if (tmp != SIV) {
    mh ← zeros(64)
    mh.State ← ace_state_import_auth
    Clear all Contents and possibly release their memory
    return
  }

  // Now the content can be deserialized.
  // This includes computing any derived values not saved as part of the SCC,
  // as well as creating shares for threshold implementations.

  // FINALLY, restore the State from the SCC
  mh.State ← saved_SN;
}
----

// ////////////////////////////////////////////////////////////////////
