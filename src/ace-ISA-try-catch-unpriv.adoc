[[ACE-horizontal-exception-handling]]
== Unprivileged Try/Catch Architecture

_This chapter is currently informative and non-normative._

The present chapter described the `Ztac` extension, which provides a
horizontal trapping mechanism to implement a try/catch mechanism for errors.

ACE has been architected so that in properly written code, the most common
executed instructions, namely `ace.exec` and `ace.state`, rarely have to deal
with errors. Errors should occur mostly with configuration operations, caused
by insufficient CRF capacity or while authenticating SCCs.

However, revocation of localities or reached expiration dates may cause a CR to
suddenly become invalid just before executing an `ace.exec` or `ace.state`
instruction. Requiring software to check for errors after each operation would
impose significant burden on the software and possibly degrade performance.

A try/catch mechanism is implemented as follows. At the
start of a subroutine that performs a cryptographic protocol, the address of
the ÔÇťcatchÔÇŁ code is written in CSR `acetacvector`
(<<ACE-CSR-acetacvector>>) and the mask with the error types to catch is
loaded in CSRs `acetacmask` (<<ACE-CSR-acetacmask>>).
Before exiting the routine, the two CSRs are cleared.

=== CSRs

`Ztac` provides two unprivileged CSRs for each feature that relies on it.
We define here only the CSRs that would be used with ACE.

[WARNING]
`Ztac` may resurrect two CSRs from the N extension, namely `uepc` and `ucause`, in case this is requested during the ratification process.


[[ACE-CSRs-trycatch]]
.ACE Unprivileged CSRs.
[cols="^10%,^10%,<18%,<62%"]
[float="center",options="header"]
|===
|  Address | Privilege | Field          | Description
|  0xXXX   | URW       | `uepc`         | (?) Address of the instruction that caused the error.
|  0xXXX   | URW       | `ucause`       | (?) Cause of the error.
|  0xXXX   | URW       | `acetacmask`   | Mask to determine which ACE exceptions horizontally trap.
|  0xXXX   | URW       | `acetacvector` | Address of the horizontal ACE trap handler.
|===


[[ACE-CSR-acetacmask]]
==== `acetacmask`

{empty}
(((CSR, `acetacmask`)))
Mask that determines which ACE exceptions horizontally trap. If bit __i__ of
this CSR is set, then ACE error __i__+1 triggers a horizontal trap, as long as
`acetacvector` is not NULL. If bit __i__ of this CSR is clear, then ACE error
__i__+1 shall not trigger a horizontal trap, but it may still trigger a
vertical trap.

_Included in_::
[%header,cols="4,2,2"]
|===
|Extension | Minimum version | Lifecycle state
| `Ztac` | v0.0.0          | --
|===

[[ACE-CSR-acetacvector]]
==== `acetacvector`

{empty}
(((CSR, `acetacvector`)))
Address of the horizontal ACE trap handler. If this pointer is NULL, horizontal
traps on ACE exceptions are disabled, regardless of the value of
`acetacmask`.

_Included in_::
[%header,cols="4,2,2"]
|===
|Extension  | Minimum version | Lifecycle state
| `Ztac` | v0.0.0          | --
|===


[WARNING]
Further pairs of CSR can be defined by other extensions to in order to be
supported by this try/catch mechanism.

=== Discussion

The trap is precise, i.e., all instructions before the one that causes the trap
are completed, and the address of the instruction that caused the error is placed
in the User EPC `uepc` CSR if available, together with the cause in `ucause`.
For try/catch mechanisms, a local variable can also be used to keep track
of the place where the error occurred, potentially eliminating the need for
CSRs `uepc` and `ucause`.  Since `aceerror` provides the cause, a CSR like
`ucause` may be considered unnecessary as well.

NOTE: in ACE, The main application of such code is when revocation or expiration
cause a CR to become invalid. In such contexts, it is not the duty of the
cryptographic subroutine to resolve the issue, but of the code that makes use
of the latter, as the keys may have to be renegotiated, and this may even need
user intervention.

////
. At the start of a subroutine that performs a cryptographic protocol, the
  address of the ÔÇťcatchÔÇŁ code is written in CSR `acetacvector`
  (<<ACE-CSR-acetacvector>>) and the mask with the error types to catch is
  loaded in CSRs `acetacmask` (<<ACE-CSR-acetacmask>>).

. The code does not directly handle the exceptions to be caught, usually only
  ace_err_invalid or ace_err_expired.
. If the code runs without such errors, then the two CSRs are cleared at the
  end of the routine before returning.
. If, say, an ace_err_invalid_CR or ace_err_expired_CR error is raised, the
  horizontal trap diverts the control flow to the catch code, which usually
  cleans up, and returns to the caller of the subroutine, not to the routine
  itself, with an error code.
////
