== Executive Summary

Cloud platforms are now the backbone of global digital infrastructure.
From AI model deployment and financial transactions to storage and communications, cryptography is embedded in nearly every critical workload.
At the same time, the sensitivity of data and intellectual property has never been higher.
In the same interconnected world, protection of user data is also paramount on client devices, since the latter can be subject to attacks from malicious actors on the internet, exploiting vulnerabilities in browsers and other software to elicit sensitive user data.

The confidentiality, integrity, and authenticity of cryptographic keys is particularly important in all these contexts.
The urgency of protecting these keys and the need for high-performance execution create a fundamental tension:
strong protection of cryptographic secrets comes at the cost of performance, while high-performance execution often exposes sensitive values to risk by making them visible to software.

This white paper analyzes this problem and outlines an architectural direction—exemplified by the concept of cryptographic contexts—that enables high-performance cryptographic execution while preserving the confidentiality and integrity of keys and all derived sensitive values.


== Why This Matters Now

=== Cryptography in Performance-Critical Software

The use of isolated security modules for cryptographic operations guarantees a high level of protection of cryptographic secrets.
However, this often comes at a high performance cost.
Therefore, cryptographic operations are often implemented in the firmware or software that requires them, either directly or through the use of special instructions.

This affects

* Secure communications;
* Content protection (e.g., DRM, model decryption);
* Secure enclaves and attestation; and
* Cloud and edge computing pipelines.

These use cases demand low-latency, high-throughput cryptography tightly integrated with general-purpose compute.

=== The Exposure Problem

Traditional software implementations expose keys and intermediate values in memory.
This happens also is specialized instructions are used that compute the cryptographic operation directly in the CPU.
This enables serious attacks, for instance through memory disclosure vulnerabilities, or cold boot attacks.

== The Limits of Existing Approaches

To protect cryptographic keys and sensitive data, existing systems typically rely on isolation. This isolation may take the form of trusted execution environments, secure enclaves, or dedicated hardware blocks designed to handle cryptographic operations. These mechanisms are effective in ensuring that secrets are not exposed to general-purpose software.

However, this protection comes at a price.

Requiring switches to protected software environments or hardened hardware blocks for such operations provides the necessary security but also heavily affects performance. Each transition into a protected domain introduces overhead, disrupts execution flow, and limits opportunities for parallelism. In large-scale cloud systems, where millions of cryptographic operations may occur every second, these costs accumulate rapidly.

The alternative, namely to implement cryptography in the general execution environment, possibly aided by acceleration instructions, solves the performance problem, but exposes the key to memory.

Intel's KeyLocker and IBM's Protected Key functionalities address this security issue by using encrypted keys instead of their actual values.
The hardware decrypts them when they are used: the values may be set by software, for instance a TEE, or hardware, but are never revealed by the hardware itself.

This leaves a more subtle but equally serious problem unaddressed:
Only hiding the keys does not necessarily ensure that the desired cryptographic guarantees cannot be broken.

Complex modes of operation require the confidentiality and integrity also of derived keys and other internal state (e.g., chaining values, counters, masks, intermediate authentication values).
Knowledge of said values may facilitate breaking these modes, or even lead to complete compromises, allowing attacker to either violate confidentiality or forge their own messages, regardless of whether the attacker can recover the key.
This applies to modes such as GCM-SIV, OCB, and also the upcoming _accordions_ that will be standardized by the NIST.

Furthermore, current cryptographic accelerators may be configured to work not only with a secret key, but also a set initialization vector, to reduce misuse.
Current “Key-Hiding Only” ISAs do not provide such an option.

In effect, protecting only the initial key while leaving the rest of the computation visible is comparable to locking the front door while leaving the windows open. It creates a false sense of security without addressing the full scope of the problem.

Finally, it is customary to define new instructions for each new cryptographic algorithm.
It is clear that this will lead to a proliferation of instructions over time, as various state actors define new standards.
In modern RISC architectures with a single or few instruction lengths, encoding space comes at a premium and defining new instructions requires a thorough review.

_Therefore, we need a solution that limits encoding space usage–ideally, a constant number of instructions, regardless of how many algorithms are supported._

== A Solution

=== Architectural Insight: Cryptographic Contexts

We suggest to treat cryptographic execution as stateful, encapsulated objects rather than stateless primitive invocations.

We achieve this by defining _Cryptographic Contexts_, which extend the notion of encrypted keys by including additional information.
A Cryptographic Context also binds its key(s) to a specific algorithm, in order to prevent key misuse.
It also contains any derived material, intermediate values, a representation of the current state of a cryptographic algorithm, and usage policies and constraints.

Cryptographic Contexts reside in dedicated architectural storage, which we call _Context Registers_.
These are not addressable as normal memory, not directly readable by software, and managed only via instructions that follow very strict rules.

=== Atomic Cryptographic Operations

Operations are executed atomically, such as full AES block encryption/decryption, steps in modes of operation, and hash absorption rounds.
This prevents leakage from revealing the outputs of intermediate steps.

=== Secure Lifecycle of Cryptographic Contexts

==== Provisioning

Cryptographic Contexts are initialized by writing two pieces of data into them:

* Secret material, including one or more keys, but also other data required by the algorithm; and
* The metadata that specifies which algorithm can use the key(s), and also defines its usage policies.

Writing these two pieces into a Context is called “provisioning”.
Provisioning binds them into a single inseparable entity.

==== Execution

Operations reference contexts rather than keys:

* No key material appears in registers or memory; and
* All computation remains inside the protected domain.

==== Export and Import

Contexts can be externalized as sealed cryptographic contexts:

* Encrypted and authenticated representations
* Bound to hardware, locality, or system keys
* Safe for storage or migration

=== State Management

Contexts maintain internal state machines:

* Support for streaming and multi-step algorithms
* Protection of intermediate values across steps


== Advantages and Challenges

=== Security Advantages

This model provides:

* End-to-end confidentiality: keys and derived values never exposed
* Integrity guarantees: metadata and state are protected
* Misuse resistance: policies restrict operations and contexts
* Side-channel mitigation: supports constant-time and masked implementations
* Composability: primitives and modes treated uniformly


=== Performance Advantages

* Elimination of enclave or accelerator switching overhead
* Tight integration with CPU pipelines
* Parallelism across multiple contexts
* Reduced memory traffic for sensitive data
* Support for vectorized or buffered data paths


=== Interoperability Challenge

While the architectural direction is clear, a major barrier remains:

* Different instruction set architectures (ISAs) define different formats
* Incompatible representations of protected keys and contexts
* Divergent semantics for cryptographic operations

This fragmentation prevents:

* Portability across heterogeneous systems
* Migration of protected workloads
* Standardized software stacks


== Call for Action

To unlock the full potential of high-performance confidential cryptography, the industry must converge on common standards.

=== Standardized Wrapped Key Format

We need:

* A common, architecture-neutral format for sealed/wrapped keys and contexts
* Cryptographic binding to policies and metadata
* Portability across devices, vendors, and environments

=== Common Semantics for Cryptographic Contexts

We must define:

* A shared specification of cryptographic context semantics
* Standardized behavior for:
** Algorithms and modes
** State machines
** Key derivation
** Policy enforcement

This ensures:

* Consistent security guarantees
* Cross-platform correctness
* Interoperability in heterogeneous deployments

=== Cross-ISA Collaboration

Vendors, standards bodies, and the research community should collaborate to:

* Align ISA extensions
* Define ABI-level contracts for cryptographic contexts
* Ensure compatibility across CPUs, accelerators, and virtualized environments


== Conclusion

The demand for secure, high-performance cryptography is rapidly increasing, driven by modern computing workloads. Existing approaches force a trade-off between security and performance that is no longer acceptable.

Architectural support for cryptographic contexts offers a path forward: enabling software to perform high-throughput cryptographic operations while ensuring that keys and all sensitive derived values remain protected.

However, realizing this vision at scale requires standardization—in formats, semantics, and interfaces. Only through a coordinated effort can we achieve secure, interoperable, and high-performance cryptographic execution across the heterogeneous computing landscape.
