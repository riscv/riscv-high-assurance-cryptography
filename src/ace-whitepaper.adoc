= White Paper. Confidential High-Performance Cryptography: A Strategic Imperative for Secure and Scalable Cloud and Client Computing
include::../docs-resources/global-config.adoc[]

== Executive Summary

Cloud platforms are now the backbone of global digital infrastructure.
From AI model deployment and financial transactions to storage and communications, cryptography is embedded in nearly every critical workload.
At the same time, the sensitivity of data and intellectual property has never been higher.
In the same interconnected world, protection of user data is also paramount on client devices, since the latter can be subject to attacks from malicious actors on the internet, exploiting vulnerabilities in browsers and other software to elicit sensitive user data.

The confidentiality, integrity, and authenticity of cryptographic keys is particularly important in all these contexts.
The urgency of protecting these keys and the need for high-performance execution create a fundamental tension:
strong protection of cryptographic secrets comes at the cost of performance, while high-performance execution often exposes sensitive values to risk by making them visible to software.

This white paper analyzes this problem and outlines an architectural direction—exemplified by the concept of cryptographic contexts—that enables high-performance cryptographic execution while preserving the confidentiality and integrity of keys and all derived sensitive values.


== Why This Matters Now

=== Cryptography in Performance-Critical Software

To attain high performance, cryptographic operations are often not isolated into security modules, but are directly embedded in user-loadable software.
This affects

* Secure communications;
* Content protection (e.g., DRM, model decryption);
* Secure enclaves and attestation; and
* Cloud and edge computing pipelines

These use cases demand low-latency, high-throughput cryptography tightly integrated with general-purpose compute.

=== The Exposure Problem

Traditional software implementations expose keys and intermediate values in:

* General-purpose registers;
* Memory (including caches); and
* Debug and virtualization layers.

This exposure enables attacks such as:

* Side-channel leakage (timing, power, cache);
* Memory disclosure vulnerabilities; and
* Fault injection and speculative execution attacks.

== Limitations of Existing Approaches

=== Protected Software Environments and Hardened Hardware

Trusted Execution Environments (TEEs), enclaves, and dedicated cryptographic accelerators provide strong protection by isolating sensitive material.

However:

Requiring switches to protected software environments or hardened hardware blocks for cryptographic operations provides the necessary security but also heavily affects performance.

The performance penalties arise from:

* Context switches across protection domains
* Serialization of execution
* Limited parallelism and bandwidth
* Data movement overhead between domains

As a result, these solutions are often unsuitable for high-throughput or latency-sensitive workloads.

=== “Key-Hiding Only” Approaches

Some architectures attempt to improve security by hiding keys in special registers or hardware compartments while still exposing computation steps.
Examples include Intel's KeyLocker and IBM's Protected Key functionalities.

This approach is insufficient.

Only hiding the keys does not necessarily ensure that the cryptographic guarantees needed by the features relying on those keys cannot be broken.
Complex modes of operation require the confidentiality and integrity also of derived keys and other internal state (e.g., chaining values, counters, masks, intermediate authentication values).
Knowledge of said values may facilitate breaking these modes, or even lead to complete compromises, allowing attacker to either violate confidentiality or forge their own messages.
This applies to modes such as GCM-SIV, OCB, and also the upcoming _accordions_ that will be standardized by the NIST.

== Architectural Insight: Cryptographic Contexts

A more robust approach is to treat cryptographic execution as stateful, encapsulated objects rather than stateless primitive invocations.

=== Cryptographic Contexts (CCs)

A cryptographic context encapsulates:

* Secrets: keys and derived material
* State: intermediate values, mode state machines
* Metadata: algorithm, policies, usage constraints

Key properties:

* Indivisible structure (cannot be partially accessed)
* Strong confidentiality and integrity guarantees
* Policy-bound usage (algorithm, privilege, locality, expiration)

=== Hardware-Enforced Isolation Without Context Switching

Contexts reside in dedicated architectural storage (e.g., context registers):

* Not addressable as normal memory
* Not directly readable by software
* Operated on only via controlled instructions

This enables:

* In-place, high-performance execution
* No need for domain switching
* Reduced attack surface

=== Atomic Cryptographic Operations

Operations are executed atomically at the algorithm or mode level:

* Full AES block encryption/decryption
* AEAD processing steps
* Hash absorption rounds

This prevents leakage from intermediate steps and mitigates oracle-based attacks.


== Secure Lifecycle of Cryptographic Contexts

=== Provisioning

Contexts are initialized with:

* Metadata (algorithm, policies)
* Secret material

Provisioning binds these into a single protected entity.

=== Execution

Operations reference contexts rather than keys:

* No key material appears in registers or memory
* All computation remains inside the protected domain

=== Export and Import

Contexts can be externalized as sealed cryptographic contexts:

* Encrypted and authenticated representations
* Bound to hardware, locality, or system keys
* Safe for storage or migration

=== State Management

Contexts maintain internal state machines:

* Support for streaming and multi-step algorithms
* Protection of intermediate values across steps


== Security Advantages

This model provides:

* End-to-end confidentiality: keys and derived values never exposed
* Integrity guarantees: metadata and state are protected
* Misuse resistance: policies restrict operations and contexts
* Side-channel mitigation: supports constant-time and masked implementations
* Composability: primitives and modes treated uniformly


== Performance Advantages
* Elimination of enclave or accelerator switching overhead
* Tight integration with CPU pipelines
* Parallelism across multiple contexts
* Reduced memory traffic for sensitive data
* Support for vectorized or buffered data paths


== Interoperability Challenge

While the architectural direction is clear, a major barrier remains:

* Different instruction set architectures (ISAs) define different formats
* Incompatible representations of protected keys and contexts
* Divergent semantics for cryptographic operations

This fragmentation prevents:

* Portability across heterogeneous systems
* Migration of protected workloads
* Standardized software stacks


== Call for Action

To unlock the full potential of high-performance confidential cryptography, the industry must converge on common standards.

=== Standardized Wrapped Key Format

We need:

* A common, architecture-neutral format for sealed/wrapped keys and contexts
* Cryptographic binding to policies and metadata
* Portability across devices, vendors, and environments

=== Common Semantics for Cryptographic Contexts

We must define:

* A shared specification of cryptographic context semantics
* Standardized behavior for:
* Algorithms and modes
* State machines
* Key derivation
* Policy enforcement

This ensures:

* Consistent security guarantees
* Cross-platform correctness
* Interoperability in heterogeneous deployments

=== Cross-ISA Collaboration

Vendors, standards bodies, and the research community should collaborate to:

* Align ISA extensions
* Define ABI-level contracts for cryptographic contexts
* Ensure compatibility across CPUs, accelerators, and virtualized environments


== Conclusion

The demand for secure, high-performance cryptography is rapidly increasing, driven by modern computing workloads. Existing approaches force a trade-off between security and performance that is no longer acceptable.

Architectural support for cryptographic contexts offers a path forward: enabling software to perform high-throughput cryptographic operations while ensuring that keys and all sensitive derived values remain protected.

However, realizing this vision at scale requires standardization—in formats, semantics, and interfaces. Only through a coordinated effort can we achieve secure, interoperable, and high-performance cryptographic execution across the heterogeneous computing landscape.
